# TajGo v1 — Courier Onboarding & Verification Spec

Заявка, проверка и допуск курьера. Заменяет текущее поведение «вошёл в
курьерский экран → профиль курьера создался сам» (ensureCourierProfile без
проверки). Статусы и данные — согласованы с AUTH_ACCOUNT_MODES_UX_SPEC §G.

---

## A. Экран «Стать курьером»

Открывается из профиля/главной («Зарабатывать с TajGo»). Продающий, но
честный:

- 🕒 «Свободный график — выходите на линию, когда удобно»;
- 🗺 «Маршруты и навигация прямо в приложении»;
- 💰 «Доход за заказ виден ДО того, как вы его приняли»;
- 🛡 «Короткая проверка — для безопасности клиентов и вашей репутации».

Кнопка: «Заполнить анкету — 3 минуты».

## B. Пошаговая анкета (5 шагов, прогресс сверху)

**Шаг 1 — Основные данные:** имя (предзаполнено из профиля), дата рождения
(поле есть, в v1 необязательное — станет обязательным по требованиям
позже), фото профиля (см. примечание §B-фото).

**Шаг 2 — Транспорт:** пешком 🚶 / велосипед 🚲 / электровелосипед 🛵 /
скутер 🛵. Автомобиль — заблокированная опция «скоро» (после v1).

**Шаг 3 — Документы:** фото паспорта/удостоверения, фото транспорта (для
вело/электро/скутера), чекбоксы согласий («данные используются только для
проверки»).

**Примечание про фото (шаги 1 и 3):** загрузка фото требует Firebase
Storage, которого в проекте пока нет. Поведение v1:
- если Storage включён владельцем (feature flag `storageEnabled`) — обычная
  загрузка с камеры/галереи;
- если нет — шаг показывает: «Фото пока не нужны — покажете документы при
  личной встрече» и пропускается без блокировки заявки. Поля URL в данных
  зарезервированы. Пилот по GTM-правилам и так «только доверенные курьеры,
  которых знаем лично» — личная встреча заменяет фото.

**Шаг 4 — Контакты и город:** телефон показан (уже подтверждён, не вводить),
город: Худжанд (фиксировано), зона работы: «Весь город» / выбор районов
(чипы из справочника районов).

**Шаг 5 — Проверка данных:** итоговая карточка всех ответов, кнопка
«Отправить заявку». После отправки — экран Pending (§D).

## C. Сохранение черновика

- Любой шаг можно покинуть — анкета сохраняется (courierStatus = draft,
  данные в courier_applications/{uid});
- при возврате: «Продолжить анкету» с того же шага;
- черновик живёт без срока давности.

## D. Pending — «Заявка на проверке»

Экран статуса:

> **Заявка на проверке 🕐**
> «Мы проверяем данные — обычно это быстро. Как только закончим, здесь
> появится результат, а пока вы можете пользоваться TajGo как клиент.»

Без ложных обещаний точного времени («в течение часа» — не писать).

Действия: «Перейти в режим клиента» (главное) · «Мои данные» (read-only
просмотр отправленного) · исправления — только если админ вернул заявку
(§F). Карточка-напоминание «Заявка на проверке» видна в клиентском профиле.

## E. Approved

Триггер — админ нажал «Одобрить» (§H). Система (транзакция):
- courierStatus = approved;
- roles += "courier";
- создаётся/обновляется couriers/{uid} (существующий ensureCourierProfile —
  переиспользовать) + courier_public/{uid};
- заявка помечается approved + resolvedBy/resolvedAt.

Пользователь при следующем входе/обновлении видит:

> **Вы приняты! 🎉**
> «Курьерский режим открыт. Пройдите короткое знакомство — 1 минута.»

→ onboarding §J → CourierHome.

## F. Rejected

Человеческий текст, без канцелярита:

> **Пока не можем принять заявку**
> Причина: {текст админа, обязателен}
> «Исправьте и отправьте снова — мы посмотрим ещё раз.»
> [Исправить и отправить снова]

Кнопка возвращает в анкету с сохранёнными данными (draft), правится нужное,
повторная отправка → pending. Количество попыток не ограничиваем в v1.

## G. Suspended

- Клиентский доступ полностью сохраняется;
- выход на линию запрещён: экран курьера показывает
  «Курьерский режим приостановлен. Причина: {текст}. Свяжитесь с нами:
  {контакт}»;
- активная трансляция координат прекращается, activeOrderId разбирается
  админом до/при suspend;
- снять suspend может только админ (→ approved).

## H. Админская проверка (новый раздел админки «Заявки курьеров»)

- Список pending-заявок (бейдж с числом на Dashboard);
- карточка заявки: профиль, телефон, транспорт, зона, фото/пометка «личная
  встреча», дата подачи, история предыдущих решений;
- действия: **Одобрить** · **Отклонить** (причина обязательна) ·
  для approved-курьеров в разделе «Курьеры»: **Приостановить** (причина) /
  **Вернуть на линию**;
- каждое действие — confirm-диалог + запись в admin_logs (action:
  approveCourier / rejectCourier / suspendCourier / reinstateCourier,
  courierId, adminId, details, createdAt) — той же транзакцией, по образцу
  существующих admin actions.

## I. Безопасность

1. Заявка и документы живут в `courier_applications/{uid}` — читает только
   владелец заявки и admin. В courier_public НИЧЕГО из заявки не попадает.
2. Телефон, дата рождения, документы клиенту не видны никогда — клиент
   видит только courier_public (имя, рейтинг, транспорт, позиция).
3. Approve/reject/suspend — только admin (`role == "admin"`), проверяется
   правилами.
4. Черновик правил для courier_applications подготовить в локальном
   firestore.rules; **deploy — только после явного разрешения владельца**
   (как всегда).
5. Фото (когда включатся) — в Storage с правилами «владелец пишет, admin
   читает»; публичных URL нет.

## J. Онбординг курьера после одобрения (4 экрана-карточки)

1. **«Выходите на линию»** — тумблер, «📡 координаты передаются», заказы
   приходят сами.
2. **«Принимайте заказ»** — карточка: маршрут, доход виден до принятия,
   один активный заказ за раз.
3. **«Карта ведёт вас»** — A→B, «Я на месте» → «Забрал» → «Передал», GPS
   должен быть включён, подтверждения работают только рядом с точкой.
4. **«Код — это деньги»** — передача по коду клиента, деньги начисляются
   после подтверждения; правила безопасности (вежливость, звонок за 5 минут
   при просьбе в комментарии).

Кнопки: «Дальше» ×3 → «Выйти на линию 🟢». Пропустить можно; повторно
доступен из профиля курьера («Как это работает»).
