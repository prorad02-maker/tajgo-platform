# TajGo v1 — Acceptance QA

Приёмочный план продукта v1. Крупные сквозные сценарии, не мелочные тесты
кнопок. Карта проверяется отдельно по MAP_NAVIGATION_MASTER_QA — здесь
только пересечения. Формат каждого теста: Сценарий · Предусловия · Шаги ·
Ожидаемо · Критический баг · Можно после пилота.

---

## 1. Новый клиент

**Сценарий:** человек впервые ставит TajGo и заказывает доставку.
**Предусловия:** чистое устройство, release-сборка, тестовый номер Phone Auth.
**Шаги:** запуск → Splash → экран намерения → «Заказать доставку» → телефон
→ SMS → имя → CustomerHome → заказ через карту → отслеживание.
**Ожидаемо:** экран намерения без слова «роль»; SMS-flow по AUTH §B; после
имени — сразу главная; заказ создаётся ≤90 сек; трекинг открылся сам.
**Критический баг:** повторный показ экрана намерения; создание заказа без
подтверждённого телефона в release; двойной UID.
**После пилота:** фото профиля, избранные адреса.

## 2. Клиент становится курьером

**Сценарий:** существующий клиент подаёт заявку и получает courier mode.
**Предусловия:** авторизованный клиент с историей заказов; admin-аккаунт.
**Шаги:** Профиль → «Зарабатывать с TajGo» → анкета 5 шагов (закрыть на
шаге 3 → вернуться) → отправить → pending → админ одобряет → пользователь
обновляет экран.
**Ожидаемо:** UID НЕ меняется; черновик сохранился; в pending клиентский
режим полностью работает; после approve — «Вы приняты», онбординг,
переключатель режимов в профиле; история заказов клиента на месте.
**Критический баг:** второй UID/повторная SMS; потеря черновика; заявка
видна другим пользователям; courier mode до одобрения.
**После пилота:** загрузка фото документов (если Storage не включён).

## 3. Новый пользователь сразу выбирает курьера

**Сценарий:** первый запуск → «Зарабатывать с TajGo».
**Шаги:** намерение → телефон/SMS → анкета → отправить → pending.
**Ожидаемо:** аккаунт создан с roles=["customer"], courierStatus=pending;
доступен клиентский режим («Пока проверяем — можно заказывать»); карточка
статуса заявки видна.
**Критический баг:** тупик «ждите» без клиентского режима; roles без
customer.
**После пилота:** —

## 4. Approved courier — полный рабочий цикл

**Сценарий:** одобренный курьер работает день.
**Предусловия:** approved; второй телефон — клиент; точки рядом (гео-гейт).
**Шаги:** переключение в курьера → онбординг → на линию → принять заказ →
навигация A → «Я на месте» → «Забрал» → B → код → completed → переключиться
в клиента → заказать самому себе доставку от другого клиента (проверка
режимов).
**Ожидаемо:** переключение без SMS; статистика выросла (earnings, orders);
lastMode=courier сохранился после перезапуска; курьер-как-клиент создаёт
заказ, но НЕ видит его в своей курьерской ленте как исполняемый
(собственные заказы курьеру не предлагаются — или явно помечены; решение
v1: свои заказы в ленте скрыты).
**Критический баг:** второй заказ при незавершённом; деньги не на completed;
собственный заказ принят самим собой; переключение режимов сбросило线线
онлайн-статус без снятия с линии.
**После пилота:** статистика за неделю/месяц.

## 5. Admin

**Сценарий:** владелец обрабатывает заявку и управляет днём.
**Шаги:** admin-аккаунт → Dashboard (бейдж заявок) → Заявки → открыть →
одобрить одну, отклонить другую с причиной → Заказы → детали → «Вернуть в
waiting» → Карта города → тест 5 секунд.
**Ожидаемо:** обычный пользователь админку не видит (проверить с
не-admin!); каждое действие — confirm + запись в admin_logs; отклонённый
курьер видит причину; возвращённый заказ появился в лентах.
**Критический баг:** admin-вход у обычного пользователя (в т.ч. в release
через kDebugMode-лазейку); действие без лога; approve не открыл courier
mode.
**После пилота:** экран просмотра admin_logs в приложении.

## 6. Возвращающийся пользователь

**Сценарий:** повторные запуски всех типов пользователей.
**Шаги:** закрыть/открыть приложение клиентом, курьером (lastMode=courier),
pending-заявителем.
**Ожидаемо:** каждый попадает в свой lastMode без экрана намерения и без
телефона; pending видит клиентский режим + карточку заявки.
**Критический баг:** повторный запрос SMS при живой сессии; экран намерения
у существующего профиля; потеря lastMode.
**После пилота:** —

## 7. Ошибки

**Сценарий:** отказы внешних систем не ломают продукт.
**Шаги/ожидаемо (каждый пункт — отдельная проверка):**
- Phone Auth disabled → «Вход по SMS временно недоступен», приложение живо;
- SMS failure/timeout → текст из AUTH §I, «Отправить ещё раз» работает;
- offline при заказе → человеческий текст, данные не теряются, после сети
  всё догоняет;
- GPS denied/denied forever → тексты + «Настройки», клиентский заказ
  возможен пином без GPS;
- routing fallback → прямая линия + «Маршрут предварительный», заказ и
  навигация живы;
- Firestore permission denied (правила строже клиента) → человеческое
  «не хватает прав», НЕ красный stack trace;
- account conflict (телефон занят) → экран из AUTH §J, никаких дубликатов
  и потерь.
**Критический баг:** stack trace/английский текст на экране; блокировка
всего приложения одной ошибкой; авто-слияние аккаунтов.
**После пилота:** ретраи с backoff, офлайн-очередь.

## 8. Миграция старого пользователя

**Сценарий:** данные, созданные до v1, продолжают работать.
**Предусловия:** в базе: (а) users с legacy role='customer' без roles;
(б) anonymous-пользователь с заказами; (в) существующий couriers/{uid} с
историей и заработком; (г) активный заказ в статусе pickedUp.
**Шаги:** вход каждым из них в v1-сборку.
**Ожидаемо:** (а) roles мигрируют при первом входе, ничего не спрошено
заново; (б) anonymous в debug работает как раньше, в release — мягкий
баннер «войдите по телефону», старые заказы видны; (в) легаси-курьер
получает courierStatus=approved АВТОМАТИЧЕСКИ (без повторной анкеты!),
статистика цела; (г) активный заказ доживает до completed по старой схеме.
**Критический баг:** потерянный заказ; легаси-курьер, отправленный на
повторную проверку; обнуление earnings/ordersToday; двойные couriers-доки.
**После пилота:** массовая серверная миграция (в v1 — ленивая, при входе).

## 9. Release-проверки

**Сценарий:** релизная сборка чиста от dev-функций.
**Шаги:** `flutter build apk --release` → полный обход всех экранов +
`git log`/diff проверка.
**Ожидаемо:**
- Demo Tools и «Тест ·»-кнопки отсутствуют;
- anonymous fallback выключен (создать заказ без телефона нельзя);
- admin debug access (kDebugMode-вход) отсутствует;
- секретов в Git нет (key.properties, keystore, google-services.json —
  по принятой политике репо);
- release APK подписан release-ключом (apksigner verify), debug-подпись —
  только для внутренних smoke-тестов, наружу не уходит.
**Критический баг:** любой пункт выше провален.
**После пилота:** автоматизация проверки в CI.
