# TajGo v0.7.0 — UX/визуальное ревью перед демо

Чеклист для быстрого прохода по приложению после отчёта Codex. Формат по
каждому экрану: ✅ что проверить · ⚠️ типовые проблемы · 🔧 что поправить,
если сыро · ✏️ тексты (эталон — DEMO_RELEASE_PRODUCT_PACK §D).

Правило прохода: смотреть на телефоне (не эмуляторе), при дневном свете,
свежими глазами — как партнёр, впервые видящий приложение.

---

## 1. Splash

- ✅ Логотип по центру, не растянут; полоса загрузки анимируется, не «висит»;
  тёмная системная тема не ломает контраст; пейзаж не обрезан на длинных
  (21:9) и коротких экранах; переход на главную без «вспышки».
- ⚠️ Полоса стоит на месте при медленной сети → выглядит как зависание;
  статус-текст с сырым текстом исключения (`FirebaseException…`).
- 🔧 Если ошибка сети видна как stack trace — заменить на человеческий текст;
  если сплэш дольше 5 сек — проверить таймауты.
- ✏️ Статус: «Подключаем TajGo…»; ошибка: «Нет связи. Проверьте интернет —
  и попробуем ещё раз.»

## 2. CustomerHomeScreen

- ✅ За 3 секунды понятно «здесь заказывают доставку»; категории ровной
  сеткой, эмодзи не разъезжаются; «Стать курьером» внизу и не спорит с CTA;
  «🛠 Управление TajGo» отсутствует у обычного пользователя (проверить на
  НЕ-админском аккаунте! и в release-сборке — kDebugMode-вход скрыт);
  «Мои заказы» ≤5 строк и не выталкивает CTA за экран.
- ⚠️ Слишком длинная главная (CTA «Заказать доставку» ниже сгиба); история
  с сырыми адресами-координатами («40.28331, 69.62215 → …») выглядит
  технично.
- 🔧 Если адрес — координаты, обрезать до 1 строки с «…»; если главная
  длинная — свернуть историю до 3 строк.
- ✏️ Приветствие по времени суток + «TajGo — платформа вашего города».

## 3. NewOrderMapScreen

- ✅ Этапы «Откуда → Куда» очевидны без объяснений (пин в центре + заголовок
  вопросом); цена видна ДО подтверждения и меняется при правке; поле
  комментария явно необязательное; «Маршрут предварительный» — мелко, не
  кричит.
- ⚠️ Геокодер возвращает координаты вместо адреса (норма для Худжанда в OSM)
  — партнёру покажется поломкой; панель деталей + клавиатура = перекрытие;
  слишком много текста в панели деталей.
- 🔧 Если координаты вместо адреса — на демо выбирать точки у известных мест
  и называть их вслух; если текста много — сократить подписи до одной строки.
- ✏️ «Откуда забрать?» / «Куда доставить?» / «Доставка от 10 TJS» /
  «Цена рассчитана по расстоянию. В тестовом режиме может быть уточнена.»

## 4. OrderTrackingScreen

- ✅ Статус читается одним взглядом (заголовок + степпер согласованы);
  маркер курьера появляется ≤10 сек после принятия и движется; код — крупные
  ячейки, читается с расстояния вытянутой руки (партнёр смотрит с чужого
  телефона!); «Получил» — лаймовая, «Не получил» — второстепенная с
  подтверждением.
- ⚠️ Баннер «Курьер найден» перекрывает чип/кнопку назад; карта пустая, если
  у заказа нет координат (старые тестовые заказы); «Не получил» слишком
  заметная — случайные споры.
- 🔧 Старые заказы без координат — удалить из демо-базы; если баннер
  перекрывает — двигается вниз на 8–16 px.
- ✏️ «🔎 Ищем курьера…» + «Обычно это занимает пару минут»; «Код получения» +
  «Назовите его курьеру при получении».

## 5. CourierHomeScreen

- 🔧 Проверить, что reset-кнопка ТОЛЬКО в debug-сборке (собрать release и

## 6. CourierOrderScreen / Navigation

- ✅ Текущая цель очевидна: пунктир до неё + верхняя плашка «Едем к точке

- 🔧 Гейт должен быть warning-жёлтым с текстом-объяснением (не красным!);

  защита ✓; для 📞 — SnackBar «Звонки появятся с телефонной авторизацией»
  говорить как фичу дорожной карты.
- ✏️ Гейт близко: «Можно подтверждать — вы у точки» (зелёным); далеко:

## A. 10 самых важных UX-критериев для демо

1. Клиент создаёт заказ ≤60 секунд без единого вопроса «а что нажать?».
2. На каждом экране ровно одна очевидная главная кнопка (лаймовая).
3. Любой статус заказа читается одним взглядом: заголовок + степпер.
4. Маркер курьера у клиента появляется ≤10 сек и движется.
5. Код подтверждения читается с чужого телефона с вытянутой руки.
6. Гео-гейт звучит как защита («подойдите ближе»), а не как ошибка.
7. Админ отвечает «что происходит в городе» за 5 секунд.
8. Ни одного сырого исключения на экране — только человеческие тексты.
9. Пустые состояния объясняют, что будет дальше, а не просто «пусто».
10. Release-сборка не содержит ни одного «Тест»/debug-элемента.

## B. 10 вещей, которые нельзя показывать партнёрам сырыми

1. FirebaseException / stack trace / англоязычные ошибки на экране.
2. Адреса-координаты «40.28331, 69.62215» в карточках заказов.
3. Нулевую админку (0 заказов, 0 оборот) — подготовить историю.
4. Кнопку «Тест · Сбросить» в основном потоке показа.
5. Застрявший заказ («Сначала завершите текущий») в начале демо.
6. Permission-denied на admin-действиях — задеплоить правила ДО показа.
7. Пустую карту диспетчера без единого курьера.
8. «Ищем GPS…» дольше 15 секунд — репетировать на улице/у окна.
9. Клавиатуру, перекрывшую кнопку «Заказать».
10. Заглушку 📞 без подготовленной фразы («появится с авторизацией»).

## C. Минимальный polish перед первой демонстрацией

Если времени мало — только это, по убыванию важности:

1. **Тексты ошибок** → человеческие (§D продуктового пака), ни одного
   исключения наружу.
2. **Release-сборка без dev-элементов** (обход всех экранов).
3. **Демо-база**: 3–5 исторических заказов, курьер с ненулевой статистикой.
4. **Деплой правил** (с разрешения владельца) → admin-действия работают.
5. **Гео-гейт**: жёлтый, с текстом-защитой; зелёная подсказка у точки.
6. **Спорные заказы** визуально выделены в админ-списке.
7. Прогон DEMO_SCRIPT + «Часть 5» на реальных телефонах — дважды.

Всё остальное из ⚠️-пунктов — в бэклог после первого показа.
