# TajGo Product v1 — отчёт о реализации пути курьера

Версия: **1.2.0+27**
Объём: задача 2 — Courier Onboarding + Verification + Admin.

## Результат

В одном аккаунте сохранены два независимых режима: клиентский доступ есть всегда, а курьерский открывается только после проверки администратором и короткого onboarding. Карта, заказы, Phone Auth, Firebase-проект и Android applicationId `tj.tajgo.app` не менялись.

Путь пользователя:

```text
customer
  → Стать курьером
  → draft (5 шагов)
  → pending
  → approved / rejected
  → onboarding
  → courier mode
```

Администратор работает с тем же документом заявки и принимает решение транзакционно. Каждое решение одновременно обновляет заявку, аккаунт и журнал `admin_logs`.

## Что реализовано

### Анкета

- экран преимуществ «Стать курьером»;
- 5 шагов: данные, транспорт, документы, зона работы, проверка;
- телефон только для чтения и привязан к текущему Auth UID;
- транспорт: пешком, велосипед, электровелосипед, скутер; автомобиль отмечен как будущий вариант;
- город Худжанд, необязательные район и доступность;
- автоматическое сохранение черновика при переходе между шагами и закрытии;
- продолжение сохранённого draft;
- итоговая карточка с возможностью вернуться к нужному шагу;
- повторная отправка после rejected.

### Фото и документы

Модель и поля URL подготовлены для Firebase Storage. В этой версии Storage не настроен и production-upload не имитируется. Пользователь видит честный вариант: документы и транспорт проверяются при личной встрече. Флаг `COURIER_STORAGE_ENABLED` по умолчанию равен `false`.

### Статусы

| Статус | Доступ курьера | Доступ клиента | Поведение |
|---|---:|---:|---|
| `draft` | нет | да | анкету можно продолжить |
| `pending` | нет | да | данные доступны только для просмотра |
| `approved` | после onboarding | да | добавляется роль courier |
| `rejected` | нет | да | показывается причина и доступно исправление |
| `suspended` | нет | да | курьер принудительно offline, причина видна |

Suspension не удаляет `activeOrderId`: активный заказ должен быть разобран администратором явно, а не потерян скрытым сбросом.

### Одобрение и onboarding

Одобрение сохраняет роль `customer`, добавляет `courier`, создаёт или объединяет `couriers/{uid}` и `courier_public/{uid}` без обнуления статистики и активного заказа. До завершения onboarding переключение и выход online запрещены.

Onboarding состоит из четырёх коротких страниц:

1. выход на линию и принятие заказа;
2. GPS и маршрут A→B;
3. код подтверждения;
4. безопасность.

После завершения записываются `courierOnboardingCompleted=true` и `lastMode=courier`. При следующих запусках onboarding не повторяется; экран «Как это работает» остаётся доступен из профиля.

## Firestore

Основные документы:

```text
users/{uid}
courier_applications/{uid}
couriers/{uid}
courier_public/{uid}
admin_logs/{autoId}
```

`courier_public` получает только публичную проекцию. Телефон, документы, доход и `activeOrderId` остаются вне неё.

Локальный `firestore.rules` подготовлен и успешно компилируется в Firestore Emulator. Deploy не выполнялся. Правила не позволяют владельцу заявки самостоятельно назначить `approved`, добавить роль `courier/admin` или писать `admin_logs`.

## Совместимость

- legacy-курьер мигрируется в `approved` и не получает дубликат профиля;
- существующие `rating`, `score`, `earningsToday`, `ordersToday`, `activeOrderId` сохраняются;
- ожидающие заказы самого пользователя не показываются ему как курьеру;
- клиентский flow, карта и маршрутизация остаются без изменений;
- online/offline не равен account mode: переход в клиентский режим выключает линию, но не удаляет курьерскую роль.

## Не входит в эту версию

- реальная загрузка фото в Firebase Storage;
- OCR/автоматическая проверка документов;
- production-deploy Firestore Rules;
- уведомления администратору о новой заявке;
- серверные custom claims для admin;
- физический end-to-end тест двумя аккаунтами и двумя телефонами.
