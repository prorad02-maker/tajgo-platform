# TajGo v0.9.0 Pilot Marketplace Release — главный тест-план

Приёмка перед закрытым пилотом. Покрывает новое (роли, маркетплейс, корзина,
предложения курьеров, радиус 1000 м, внешний навигатор) и защищает старое
(карта v0.8.7 — регресс запрещён). Карта отдельно: MAP_NAVIGATION_MASTER_QA.

Обозначения: 🔴 критично (пилот не стартует) · 🟠 высокое · 🟡 среднее.

медленные изображения без плейсхолдеров. Проверка — попутно во всех
разделах; при обнаружении — списком в бэклог, релиз не блокируют
(кроме текстов из stop-листа L старых доков: stack trace = 🔴).

## B. Тест первого запуска и ролей

Основной прогон (по шагам 1–10 из ТЗ): новая установка → выбор «клиент» →
перезапуск → роль сохранена → клиентское меню (без курьерских/админ
пунктов) → смена роли в настройках → курьер → перезапуск → курьерское
меню → админ через onboarding НЕДОСТУПЕН (нет такого варианта нигде).

| Особый случай | Ожидаемо |
|---|---|
| Старый пользователь без role | не падает; мягкий выбор роли один раз; данные целы |
| Старый пользователь с role | сразу своя главная, выбор не показывается |
| Повреждённый SharedPreferences (очистить данные приложения) | как новая установка; Firestore-профиль подхватывается |
| Нет интернета на первом входе | человеческий текст, retry; не белый экран |

🔴: роль перепуталась; админ доступен из onboarding. 🟠: повторный выбор
роли у старого пользователя.

## E. Предложения курьеров (offers)

Проверить (два курьера, один клиент):

- курьер «Принять цену» → предложение = цена клиента;
- «Предложить свою» → выше клиентской (ниже — блокируется? по спеке: выше
  или равна — сверить с master spec);
- обновить своё предложение; отозвать;
- срок истёк → предложение закрывается само, у клиента исчезает;
- два курьера → у клиента список из двух с ценами/рейтингом/расстоянием;
- клиент выбирает одного → транзакция: заказ accepted этому курьеру,
  **остальные предложения закрыты** с уведомлением «Заказ забрали»;
- двойной выбор (даблтап/два устройства клиента) → один победитель;
- два клиента не могут одновременно закрепить одного курьера, если у него
  уже активный заказ (правило G побеждает — второе принятие отклонено).

🔴: выбранный ≠ закреплённый; предложение к чужому заказу проходит;
после выбора цена ≠ цене выбранного предложения.

## F. Радиус 999/1000

Правило: **distance < 1000 м → доступно; distance ≥ 1000 м → недоступно.**

| Дистанция | Ожидаемо |
|---|---|
| 0 м | доступно |
| 500 м | доступно |
| 998 м | доступно |
| **999 м** | **доступно** |
| **1000 м** | **недоступно** (текст «Вы слишком далеко — подойдите ближе») |
| 1001 м | недоступно |
| 1500 м | недоступно |

Проверка mock-локациями (эмулятор) с точным расчётом до точки A.

Дополнительно:

| Условие | Ожидаемо |
|---|---|
| GPS старше 15 сек | действие блокировано: «Обновляем позицию…», не старая точка |
| Accuracy 20 м | работает как есть |
| Accuracy 80 м | предупреждение о неточном GPS; правило по спеке (консервативно: не пропускать на границе) |
| GPS выключен | недоступно + текст с кнопкой настроек |
| Позиции нет вообще | недоступно, без краша |
| Движение через границу 1000 м | кнопка оживает при пересечении внутрь (стрим), без рестарта экрана |

🔴: 1000 м принят; устаревший GPS пропустил.

## G. Один активный заказ курьера

Последовательность: принять заказ №1 → открыть заказ №2 → «Принять»
заблокировано с текстом «Сначала завершите текущий» → «Предложить цену»
тоже заблокировано → завершить №1 (код) → №2 стал доступен →
activeOrderId == null в базе.

Особые: спорный заказ (disputed) — курьер ЗАБЛОКИРОВАН до разбора;
отменённый админом — activeOrderId очищен, курьер свободен;
admin clearActiveOrder — разблокирует, лог записан.

🔴: второй заказ принят; activeOrderId завис после completed.

## H. Внешний навигатор

Матрица: {TajGo (внутренний) · Яндекс Карты/Навигатор · Google Maps · 2GIS ·
системный выбор} × {установлено · не установлено · deep link не
поддерживается}.

Ожидаемо:
- установлено → открывается С координатами ЦЕЛИ ЭТАПА: accepted → точка A,
  pickedUp → точка B (сменить статус и снова открыть — точка сменилась!);
- не установлено → fallback (веб/системный) или понятный текст, БЕЗ краша;
- координаты в URI верные (сверить с пином вручную);
- **в URI НЕТ кода получения, телефона, имени клиента** — только
  координаты и метка «TajGo»;
- выбор навигатора сохраняется и применяется в следующий раз.

🔴: старая точка (A после pickedUp); персональные данные в URI.

## J. Товары и корзина

Операции: добавить · +1 · −1 (до нуля = удалить) · удалить · комментарий к
позиции/заказу.

Единицы: item · kg (дробные количества, 0.5 кг) · portion · bouquet —
цена × количество считается верно для каждой.

Цены: oldPrice отображается зачёркнутым, в сумму идёт актуальная;
unavailable — добавить нельзя; hidden — не виден; **изменение цены админом
при товаре в корзине** → корзина показывает новую цену с уведомлением
(или пересчёт при оформлении — по спеке, но НЕ молча).

Суммы: subtotal = Σ(цена×кол-во) · delivery отдельной строкой · total =
subtotal + delivery. Проверить вручную на 3+ позициях с kg-дробями.

Пустая корзина — дружелюбный экран, кнопка в каталог.

**Правило (зафиксировано): одна корзина — один партнёр.** Добавление товара
второго партнёра → диалог «Очистить корзину и начать заказ у {партнёр}?».
Если Codex реализовал мультикорзину — правило пересматривается, но молчаливое
смешение двух партнёров в одном заказе = 🔴.

## M. Offline / слабая сеть

| Сценарий | Ожидаемо |
|---|---|
| Авиарежим до запуска | приложение открывается, человеческий текст, retry |
| Сеть пропала после открытия | UI жив, действия ставятся/ошибка с retry |
| Пропала при публикации заказа | ошибка с повтором; **после retry — ОДИН заказ** (идемпотентность/проверка дубля) |
| Пропала при отправке предложения | одно предложение после восстановления |
| Пропала при выборе предложения | выбор либо прошёл, либо нет — не «полусостояние» |
| Route fallback | прямая + подпись, без блокировки |
| Каталог без сети | кэш или заглушка, не вечный спиннер |
| Изображения без сети | плейсхолдеры |
| Локальная роль | работает офлайн (SharedPreferences) |
| Повтор операции после восстановления | без дублей (главное — заказы и предложения) |

🔴: двойное создание заказа/предложения после retry.

## N. Firestore rules и indexes (локально, БЕЗ deploy)

Проверить чтением правил + поведением приложения:

- клиент читает только свои заказы; предложения — только своего заказа;
- курьер создаёт предложение только от своего uid и только к открытому
  заказу; менять чужие предложения не может;
- курьер не может «выбрать себя» (выбор — только клиентская/админская
  операция);
- partners и products читаются публично (signed-in), пишутся только admin;
- неизвестные коллекции закрыты (deny by default);
- indexes: новые составные запросы (предложения по заказу, каталог по
  категории+isActive) описаны в firestore.indexes.json.

**После реального deploy обязательно повторить:** полный цикл custom
delivery, полный цикл catalog-заказа, предложения двух курьеров, радиус,
admin CRUD, миграционные аккаунты — под живыми правилами (история проекта
знает случай, когда правила ломали GPS).
