# TajGo — Spec v3: «Курьер: карта, навигация и честная передача заказа»

Самодостаточное техническое задание. Перед началом прочитай `docs/ARCHITECTURE.md`
и выполненные `docs/SPEC_CODEX_V1.md` (раздел 5 — дизайн-токены) и
`docs/SPEC_CODEX_V2.md` — эта итерация строится на них.

---

## 1. Контекст

После итераций 1–2 работает: заказ создаётся клиентом через живую карту
(точки A/B — GeoPoint, расчёт км/мин/цены), курьер на линии транслирует
координаты (foreground, шаг 30 м), принимает заказ из ленты, отмечает
«Забрал» → «Доставил», заработок растёт при доставке. Клиент видит статусы
realtime. DI — `TajGoScope`, Firestore — только через репозитории.

Проблемы, которые закрывает эта итерация:

1. У курьера нет карты: после принятия заказа он видит только текст адресов.
2. «Забрал»/«Доставил» можно нажать откуда угодно — нет гео-контроля.
3. Доставка завершается словом курьера — клиент никак не подтверждает
   получение.
4. Курьер может набрать несколько заказов одновременно.

Образец UX — курьерские приложения Яндекс Еды/Доставки (стадии «еду на
точку → на месте → забрал → еду к клиенту → передал», клиент видит курьера
на карте) и механика подтверждения PIN-кодом как в Uber Eats/DoorDash.

## 2. Цель итерации

> Курьер принимает любой заказ из ленты (распределение «кто ближе» будет
> системным позже) → открывается экран заказа с картой: его живая позиция,
> точки A и B, кнопка «Навигатор» → у точки A жмёт «Забрал посылку»
> (проверка: не дальше 2 км от A) → едет к B → жмёт «Передал заказ» (проверка
> 2 км от B) и вводит код клиента → заказ завершён, заработок начислен.
> Если клиент не назвал код — заказ переходит в «ждёт подтверждения»:
> клиент в своём приложении жмёт «Получил» или «Не получил», и пока он
> не подтвердил, курьер НЕ может брать новые заказы.

## 3. Жёсткие ограничения

1. **Новые pub-пакеты: разрешён ровно один — `url_launcher`** (последняя
   стабильная) для кнопки «Навигатор». Больше ничего.
2. **Не трогать** `lib/firebase_options.dart`, `firebase.json`,
   `android/app/google-services.json`.
3. Без Bloc/Riverpod/go_router. `setState` + `StreamBuilder`, DI через
   `TajGoScope`, навигация `Navigator`.
4. Firestore/Auth — только через репозитории `lib/core/services/`.
   Прямых импортов `cloud_firestore` в фичах нет (модели — можно).
5. Серверное время — `FieldValue.serverTimestamp()`. Все критичные переходы
   статусов — транзакции.
6. Геолокация — только foreground, как сейчас.
7. UI и комментарии — русские; дизайн — токены `TajGoColors`, CTA — лайм.
8. Маршрут на карте — прямая линия (OSRM — следующие итерации).

## 4. Модель данных (дельта)

### orders/{orderId}

| Поле | Тип | Описание |
|---|---|---|
| confirmationCode | string | 4 цифры, генерируется при создании заказа |
| completedAt | timestamp | когда клиент/код подтвердил получение |
| disputedAt | timestamp | когда клиент нажал «Не получил» |

### Машина статусов (обновлённая)

```text
waiting → accepted → pickedUp → completed        (курьер ввёл верный код)
                          │
                          └→ delivered → completed  (клиент нажал «Получил»)
                                   │
                                   └→ disputed      (клиент нажал «Не получил»)
waiting → cancelled (клиент, как раньше)
```

- `delivered` теперь означает «курьер передал, ждём подтверждения клиента».
- `completed` — финальный успех. **Начисление `earningsToday` переносится
  с перехода в `delivered` на переход в `completed`** (в той же транзакции).
- `disputed` — клиент заявил, что не получил. Для MVP это тупиковый статус
  (разбор — вручную/админкой позже).

### couriers/{uid}

| Поле | Тип | Описание |
|---|---|---|
| activeOrderId | string \| null | текущий заказ курьера; правило «один заказ» |

## 5. Задачи

### Задача 1. Модель и статусы

`lib/core/models/tajgo_order.dart`:

- В `OrderStatus` добавить `completed`, `disputed` (строки `'completed'`,
  `'disputed'`).
- Поля `confirmationCode` (String?, из документа), `completedAt`,
  `disputedAt` (DateTime?).
- В `toCreateMap()` включить `confirmationCode`.

`lib/core/models/tajgo_courier.dart`: поле `activeOrderId` (String?).

### Задача 2. Генерация кода и гео-гейт (чистые функции)

В `lib/core/services/pricing.dart` (или новый `order_rules.dart` рядом):

```dart
/// 4-значный код подтверждения, '0000'..'9999' (Random, ведущие нули сохранять).
String generateConfirmationCode([Random? random]);

/// Радиус, в котором курьеру разрешено подтверждать действия у точки.
const double actionRadiusKm = 2.0;

/// Можно ли подтверждать действие на расстоянии km от точки.
bool withinActionRadius(double km) => km <= actionRadiusKm;
```

`OrderRepository.createOrder`: генерировать и записывать `confirmationCode`.

### Задача 3. Правило «один активный заказ» + новые переходы

`lib/core/services/courier_repository.dart`:

1. **`acceptOrder`** — расширить транзакцию: прочитать `couriers/{courierId}`;
   если `activeOrderId != null` → `StateError('Сначала завершите текущий
   заказ.')`. При успехе записать `activeOrderId = orderId` в документ курьера
   (та же транзакция).
2. **`markPickedUp`** — без изменений переходной логики (accepted → pickedUp),
   но добавить параметр `distanceToPointKm` (double?): если он не null и
   `!withinActionRadius(...)` — StateError «Вы слишком далеко от точки
   забора (N км). Подойдите ближе.» Проверка расстояния — ДО транзакции,
   в репозитории.
3. **Новый `completeWithCode`** — транзакция pickedUp → completed:
   параметры `orderId, courierId, code, distanceToPointKm`. Проверки:
   статус pickedUp, `courierId` совпадает, гео-гейт, `code ==
   order.confirmationCode` (если в документе кода нет — старый заказ,
   проверку кода пропустить). При успехе: `status: completed`, `completedAt`,
   `earningsToday: increment(price)`, `couriers.activeOrderId: null`.
   Неверный код → `StateError('Код не совпадает. Уточните код у клиента.')`.
4. **`markDelivered`** (курьер передал без кода) — pickedUp → delivered +
   гео-гейт. `activeOrderId` НЕ очищать, заработок НЕ начислять — заказ
   продолжает блокировать курьера до решения клиента.
5. **Новые клиентские переходы** — в `lib/core/services/order_repository.dart`:
   ```dart
   /// Клиент подтвердил получение: delivered → completed.
   /// Транзакция: += earningsToday курьеру, couriers.activeOrderId = null.
   Future<void> confirmReceived(String orderId);
   /// Клиент не получил: delivered → disputed (+ disputedAt).
   /// activeOrderId у курьера НЕ очищать — курьер остаётся заблокирован.
   Future<void> reportNotReceived(String orderId);
   ```
6. `activeCourierOrdersStream` — активными считать `accepted`, `pickedUp`,
   `delivered` (delivered блокирует). `cancelled` заказа в статусе waiting
   с назначенным... — не актуально; но если клиент отменил waiting-заказ,
   activeOrderId не задет (accept ещё не было) — ок.

### Задача 4. Экран активного заказа курьера с картой

Новый `lib/features/courier/courier_order_screen.dart`. Открывается:
тапом по карточке «Мой активный заказ» и автоматически сразу после
успешного «Принять».

**Карта** (FlutterMap, OSM-тайлы как везде):

- Пины: A — `Icons.location_pin` тёмно-зелёный, B — лаймовый (как на
  клиентской карте). Линия A→B — `TajGoColors.green`, width 4.
- Живая позиция курьера: слушать `locationService.positionStream()`
  (подписка экрана, отменять в dispose) — синяя точка, как `_CurrentDot`
  на клиентской карте. Первичная позиция — `determineCurrentPosition()`.
- Пунктирная линия от позиции курьера до текущей цели (A на этапе accepted,
  B на этапе pickedUp): `Polyline` с `pattern: StrokePattern.dashed(...)`
  (в flutter_map 8 есть паттерны), цвет `muted`.
- Камера при открытии — `CameraFit.coordinates` по [позиция, A, B].

**Нижняя панель** (белая, скругление 20 сверху) — по статусу заказа
(StreamBuilder на стрим заказа — добавить в CourierRepository
`Stream<TajGoOrder?> orderStream(String orderId)`):

- `accepted`: заголовок «Едем за посылкой», адрес A, строка
  «До точки: N км» (dist от живой позиции до A, пересчитывать на каждое
  обновление позиции), ряд кнопок: «Навигатор» (вторичная) + «Забрал
  посылку» (лайм CTA; вызывает `markPickedUp` с текущей дистанцией;
  StateError → SnackBar).
- `pickedUp`: заголовок «Везём клиенту», адрес B, «До точки: N км»,
  кнопки: «Навигатор» + «Передал заказ» (лайм). «Передал заказ» открывает
  диалог: поле для 4-значного кода + кнопка «Подтвердить» (→
  `completeWithCode`) + текстовая кнопка «Клиент не может назвать код»
  (→ `markDelivered`). Расстояние для гео-гейта передавать в оба вызова.
- `delivered`: «⏳ Ждём подтверждения клиента» + пояснение «Пока клиент не
  подтвердит получение, новые заказы недоступны». Без кнопок.
- `completed`: «✅ Заказ завершён! +N TJS» + кнопка «К заказам»
  (pop до courier_home).
- `disputed`: «⚠️ Клиент сообщил, что не получил заказ. С вами свяжутся.»

Если геолокация недоступна (отказ/ошибка) — дистанцию не считать,
в кнопках подтверждения передавать `distanceToPointKm: null`?
**Нет:** гео-гейт обязателен. При недоступной геолокации показывать
«Включите геолокацию, чтобы подтвердить действие» и дизейблить CTA.

**Кнопка «Навигатор»** — через `url_launcher`:
- Android: `geo:LAT,LNG?q=LAT,LNG(TajGo)`;
- iOS: `https://maps.apple.com/?daddr=LAT,LNG`;
- если не открылось — fallback
  `https://www.google.com/maps/dir/?api=1&destination=LAT,LNG`;
- все запуски `LaunchMode.externalApplication`, ошибки — SnackBar.

## 6. Критерии приёмки (одно устройство, эмулятор с mock-гео)

1. Клиент создал заказ → режим курьера → «Принять» → автоматически открылся
   экран заказа с картой: пины A/B, линия, синяя точка курьера, «До точки: N км».
2. Mock-позиция дальше 2 км от A → «Забрал посылку» → SnackBar «Вы слишком
   далеко...», статус не изменился. Подвинуть mock ближе 2 км → переход
   в pickedUp, панель сменилась на «Везём клиенту».
3. «Навигатор» открывает внешнее приложение карт с точкой назначения.
4. У клиента на статусе pickedUp виден 4-значный код.
5. «Передал заказ» + верный код (рядом с B) → completed: панель «✅ Заказ
   завершён», заработок вырос, курьер снова видит ленту заказов.
   Неверный код → SnackBar, статус не изменился.
6. Повторить с «Клиент не может назвать код» → delivered: курьер видит
   «Ждём подтверждения», лента заказов ЗАКРЫТА подсказкой; второй заказ
   принять нельзя (StateError при попытке). Клиент жмёт «Получил» →
   completed, заработок вырос, курьер разблокирован.
7. Вариант «Не получил» → disputed: курьер остаётся заблокирован, у клиента
   «Мы разбираемся».
8. `flutter analyze` / `flutter test` чистые.

## 7. Вне рамок (НЕ делать)

- Автораспределение заказов «кто ближе» (серверная логика, позже).
- Авто-подтверждение получения по таймауту (нужны Cloud Functions).
- Реальная маршрутизация OSRM, повороты, голосовая навигация.
- Живой маркер курьера на карте клиента (следующая итерация).
- Фото-подтверждение доставки, подпись, чат курьер-клиент.
- Экран разбора disputed-заказов (админка).
- Серверная валидация кода (сейчас код лежит в документе заказа и
  проверяется на клиенте; при внедрении security rules перенести проверку
  в Cloud Function — записать как техдолг).
- Любые пакеты, кроме `url_launcher`.
