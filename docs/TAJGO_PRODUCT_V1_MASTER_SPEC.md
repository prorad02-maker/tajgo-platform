# TajGo Product v1 — Master Spec

Единая продуктовая спецификация TajGo v1. Главный документ пакета v1;
детали: AUTH_ACCOUNT_MODES_UX_SPEC (вход и режимы),
COURIER_ONBOARDING_VERIFICATION_SPEC (заявка курьера),
PRODUCT_V1_ACCEPTANCE_QA (приёмка), CODEX_PRODUCT_V1_HANDOFF (реализация).
Карта: MAP_PRODUCT_UX_SPEC + NAVIGATION_PRO_PRODUCT_SPEC +
TAJGO_MAP_PILOT_MASTER_PLAN — остаются в силе без изменений.

Этот документ разрешает накопившиеся противоречия (см. HANDOFF §0) и
фиксирует окончательные решения v1.

---

## A. Что такое TajGo v1

Городская платформа доставки Худжанда, один продукт с тремя лицами:

- **Клиент** заказывает доставку на карте за минуту и следит за курьером.
- **Курьер** принимает заказы, едет по навигации, подтверждает передачу кодом.
- **Владелец** управляет всем через админку: заказы, курьеры, заявки, карта
  города.

**Ключевое решение v1: один человек = один аккаунт = несколько режимов.**
Клиент и курьер — НЕ разные пользователи, а режимы одного аккаунта.
Карта — центр всех основных сценариев и quality gate каждого релиза (§D).

## B. Пользовательские типы

Модель «одна роль навсегда» — отменена. Действует:

```
Один Firebase UID → один номер телефона → один профиль → несколько режимов
```

**Режимы** (переключаемые пользователем):
- `customer` — есть у каждого с момента регистрации;
- `courier` — добавляется после одобрения заявки (см. COURIER_ONBOARDING).

**Служебные полномочия:**
- `admin` — не режим, а полномочие. Назначается ТОЛЬКО вручную в Firestore
  Console (users/{uid}.role = "admin", совместимо с текущей реализацией).
  Из приложения получить admin нельзя ни при каких условиях.

Следствия: курьер может заказать себе доставку, не выходя из аккаунта;
клиент может подать заявку курьера; повторная SMS при переключении режимов
не нужна.

## C. Главная навигация продукта

**Customer mode:**
- Главная (платформа: активный заказ, «Заказать доставку», категории,
  «Мои заказы», «Зарабатывать с TajGo»);
- Создать заказ (карта, NewOrderMapScreen);
- Мои заказы (история + активный → трекинг);
- Профиль (§F).

**Courier mode:**
- Заказы (лента waiting, статус линии);
- Активная доставка (навигация, CourierOrderScreen);
- Статистика (заработок, заказы, рейтинг, score);
- Профиль (§G).

**Admin (полномочие поверх любого режима):**
- Dashboard; Заказы; Курьеры; **Заявки курьеров** (новое); Карта города.

Переключение режимов — из Профиля («Перейти в режим…»), lastMode
запоминается. Тумблер online/offline курьера НЕ является переключателем
режимов — это только статус линии.

## D. Карта как основа

Без изменений архитектуры — закрепляем достигнутое как обязательный
**quality gate каждого релиза v1+**:

- поиск коротких адресов (справочник мест + алиасы);
- выбор A/B пином с подтверждением;
- GPS-цепочка (разрешения → позиция → трансляция);
- дорожный маршрут (route provider, fallback — прямая);
- ETA без ложной точности;
- навигация курьера (follow, инструкции, reroute, arrival);
- отслеживание клиента (живой маркер, «курьер рядом»);
- карта диспетчера (город целиком, тест 5 секунд).

Gate: релиз не выходит, если MAP_NAVIGATION_MASTER_QA §A (три ворот) не
проходит. Любая новая фича, ломающая карту, откатывается.

## E. Полный жизненный цикл заказа

Решение о совместимости: **Firestore-статусы v1 = текущие** (waiting,
accepted, pickedUp, delivered, completed, disputed, cancelled).
`draft` — локальное состояние экрана создания (в Firestore не пишется).
`arrivedAtPickup` — существующий флаг `arrivedAtPickupAt` при accepted.
`arrivedAtDropoff` — новый необязательный флаг `arrivedAtDropoffAt` при
pickedUp (пишется по кнопке/арривал-детекции, ничего не блокирует).

| Состояние | Кто меняет | Клиент видит | Курьер видит | Админ видит | Разрешено дальше | Запрещено |
|---|---|---|---|---|---|---|
| draft (локально) | клиент | карту выбора точек | — | — | создать → waiting; закрыть | попадание в базу без подтверждения |
| waiting | создал клиент | «Ищем курьера» + Отменить | карточку в ленте: Принять/Отказаться | в списке «Ожидают» | accepted (курьер), cancelled (клиент/админ) | принять при активном заказе курьера; принять двумя курьерами |
| accepted | курьер (транзакция) | «Курьер найден» + маркер | навигацию к A, «Я на месте» | активный, курьер назначен | +arrivedAtPickupAt → pickedUp; возврат в waiting (админ); cancelled (админ) | pickedUp без arrivedAtPickupAt; отмена клиентом |
| accepted+arrived | курьер | «Курьер на месте забора» | «Забрал заказ» (гео-гейт 2 км) | то же | pickedUp | — |
| pickedUp | курьер (гео-гейт) | «Заказ у курьера» + КОД + ETA | навигацию к B, «Передал заказ» | активный | +arrivedAtDropoffAt; completed (код); delivered (без кода); disputed/cancelled (админ) | completed с неверным кодом |
| delivered | курьер (без кода) | «Подтвердите получение»: Получил / Не получил | «Ждём подтверждения», новые заказы закрыты | «требует внимания» | completed (клиент/админ), disputed (клиент/админ) | новый заказ курьеру |
| completed | код / клиент / админ | «Доставлено. Спасибо!» → история | «+N TJS», разблокирован | завершённый, оборот вырос | — (финал) | любые изменения |
| disputed | клиент / админ | «Мы разбираемся» | заблокирован, «доставка на проверке» | выделен, действия разбора | completed / cancelled / waiting — только админ | самостоятельный выход курьера |
| cancelled | клиент (из waiting) / админ (любой) | «Заказ отменён» | исчезает из ленты / снят | отменённый + причина | — (финал) | — |

Деньги курьеру (earningsToday, ordersToday) — только на переходе в
completed. Это правило v1, без исключений.

## F. Профиль клиента

- Имя (displayName), телефон (подтверждённый), фото — optional (после
  включения Storage);
- **Избранные адреса** (новое): «Дом», «Работа» + до 5 своих — используются
  в поиске адресов первыми;
- История заказов (есть);
- Способы оплаты — заглушка «Наличные / перевод» (встроенные платежи после v1);
- **«Зарабатывать с TajGo»** → заявка курьера (или статус заявки);
- Настройки (§I); Помощь; Выход.

## G. Профиль курьера

Приватная часть (couriers/{uid} + заявка): имя, фото, телефон, тип
транспорта (пешком/велосипед/электровелосипед/скутер; авто — после v1),
фото транспорта, документы (см. ONBOARDING §B/I), статус проверки
(courierStatus), рейтинг, ordersToday/всего, заработок, online/offline,
activeOrderId.

Публичная часть (courier_public — как сейчас): имя, рейтинг, транспорт,
позиция, online. Документы и телефон в public НЕ попадают никогда.

## H. Уведомления — product contract

| Событие | Сейчас (in-app, realtime-стримы) | Позже (push) |
|---|---|---|
| Новый заказ рядом | лента курьера обновляется сама | push курьерам online |
| Курьер найден | баннер + статус у клиента | push клиенту |
| Курьер приехал (A/B) | статус «на месте» | push клиенту |
| Заказ забран | статус + код на экране | push клиенту |
| Курьер рядом (~300 м) | плашка «приготовьте код» | push клиенту |
| Заказ завершён | экраны обеих сторон + статистика | push клиенту (чек) |
| Заявка курьера одобрена/отклонена | экран статуса заявки при входе | push заявителю |

v1 работает целиком на in-app (открытое приложение); push — инфраструктура
после v1, контракт выше — готовое ТЗ для него.

## I. Настройки

- Режим: «Перейти в режим клиента/курьера» (если режима два);
- Геолокация: статус разрешения + кнопка в системные настройки;
- Тема: системная/светлая (тёмная тема приложения — после v1; сплэш уже
  умеет обе);
- Язык: русский (таджикский — после v1, каркас строк готовить);
- Помощь (FAQ-экран, статичный);
- Политика конфиденциальности (ссылка/экран);
- Выход (см. AUTH §K).

## J. Состав TajGo v1

**Обязательно (без этого v1 не выходит):**
аккаунт (телефон, один UID) · клиентский режим · курьерский режим ·
карта-gate (§D) · полный цикл заказа (§E) · отслеживание · код получения ·
админка · заявка курьера с проверкой.

**После v1 (зафиксировано, не делаем):**
встроенные платежи · push · партнёрский кабинет · маркетплейс/каталоги ·
голосовая навигация · пробки · iOS-релиз · мультигород · тёмная тема
приложения · таджикский язык · авто как транспорт курьера.
