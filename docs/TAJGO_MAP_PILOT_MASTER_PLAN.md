# TajGo — Map Pilot Master Plan (v0.9.1 → v1.0)

Один документ о том, как довести карту TajGo до первых партнёров и пилота.
Связанные документы: MAP_NAVIGATION_MASTER_QA (тест-план),
NAVIGATION_PRO_PRODUCT_SPEC (навигация), GO_TO_MARKET_STARTER_PACK
(партнёры). Этот план — зонтик над ними: стратегия, адресная база, полевые
тесты, решение о расширении.

---

## A. Главная стратегия

TajGo не обязан побеждать Яндекс Навигатор в пробках, голосе и 3D — и не
будет пытаться. TajGo становится **лучшим инструментом доставки по
Худжанду**:

- локальные короткие адреса («Панчшанбе», а не улица-дом-корпус);
- точки партнёров — проверенные, с уточнениями («вход со двора»);
- сценарий «забрать → доставить» как одна задача с автопереключением;
- код подтверждения — маршрут заканчивается фактом передачи;
- клиент видит курьера и честный ETA;
- админ видит город целиком.

Formula: **побеждаем не картой, а знанием города и сценария.** Каждый сомони
и час вкладываем туда, где Яндекс не конкурент по определению.

## B. «Карта готова к пилоту» — измеримые критерии

| # | Критерий | Как меряем |
|---|---|---|
| 1 | Клиент создаёт заказ ≤90 сек | секундомер, человек «с улицы» (QA §N) |
| 2 | Курьер всегда знает следующую цель | вопрос «куда сейчас?» на любом этапе — мгновенный ответ |
| 3 | Курьер не открывает чужой навигатор | наблюдение за полный день теста: 0 запусков внешних карт |
| 4 | Клиент видит понятный ETA | «сколько ждать?» → ответ числом минут без листания |
| 5 | Админ понимает город за 5 сек | тест 5 секунд (QA §D) |
| 6 | Fallback без паники | 30 сек авиарежима: ни красных экранов, ни потери заказа |
| 7 | «Не найдено» — не тупик | любой провальный поиск заканчивается кнопкой «Выбрать на карте» |

Все 7 — обязательные. Проверяются в полевом тесте (§J) и в 7-дневном
пилоте (§K).

## C. Клиентский UX пилота

Flow (детали и недопустимые ошибки — QA §B):

открыть → увидеть себя → «Откуда» (моё местоположение / поиск / пин) →
«Куда» (короткое название / недавние / пин) → цена и маршрут на карте →
«Найти курьера» → отслеживание с ETA → код при получении.

Эталонные тексты (уже канонизированы, менять только через этот документ):

| Экран | Текст |
|---|---|
| Откуда | «Где забрать?» · «📍 Моё местоположение» · «Доставка от 10 TJS» |
| Куда | «Куда доставить?» · «Введите адрес, место или ориентир» |
| Цена | «Итоговая цена: N TJS» · «Цена рассчитана по расстоянию. В тестовом режиме может быть уточнена.» |
| Маршрут | «Маршрут предварительный — показана прямая линия.» (только при fallback) |
| Ожидание | «🔎 Ищем курьера…» · «Обычно это занимает пару минут» |
| В пути | «📦 Заказ у курьера» · «примерно N мин» |
| Рядом | «Курьер рядом — приготовьте код получения» |
| Код | «Код получения: 0000» · «Назовите его курьеру при получении» |
| Финал | «✅ Доставлено. Спасибо!» |

## D. Курьерский UX пилота

Flow: линия → принял → «Едем к точке забора» → доехал → «Я на месте» →
«Забрал заказ» → «Везём клиенту» → доехал → «Передал заказ» + код →
«✅ Доставлено! +N TJS».

Эталонные тексты экрана:

| Момент | На экране |
|---|---|
| На линии | «Вы на линии 🟢» · «📡 координаты передаются» |
| Карточка заказа | маршрут · «Ваш доход за заказ: N TJS.» · 💬 комментарий |
| Этап A | «Едем к точке забора · N км · ~N мин» + манёвр («Через 200 м — направо») |
| У точки | «Можно подтверждать — вы у точки» (зелёным) |
| Далеко | «⚠ Подойдите ближе, чтобы подтвердить действие» (жёлтым) |
| Этап B | «Везём клиенту · N км · ~N мин» + 💬 комментарий клиента |
| Передача | диалог «Код клиента» · «Подтвердить» |
| Финал | «✅ Доставлено! +N TJS» |
| GPS/сеть | «GPS неточный — лучше всего сигнал на улице» · «Связь слабая. Позиция видна, отправим при появлении сети.» |

## E. Админский UX пилота

- **Dashboard**: заказы сегодня · активные · завершённые · спорные · курьеры
  online · оборот. Тап по цифре → список с фильтром.
- **Карта города**: открывается на весь Худжанд; курьеры (зелёные со
  стрелкой), waiting (тёмные пины), активные (пара A/B); фильтры-чипы;
  сводка-пилюля.
- **GPS давно не обновлялся**: в списке курьеров «координаты обновлены N мин
  назад», > 10 мин — жёлтым, > 30 мин — красным. На пилоте это главный
  сигнал «курьер потерялся/телефон сел».
- **Selected order**: тап по любому маркеру/заказу → peek-карточка → детали
  с таймлайном и действиями (вернуть в waiting, завершить вручную...).
- Спорные — выделены цветом всегда, им нет «срока давности».

## F. Адресная база Худжанда — первые 100 точек

### Схема точки (одна запись справочника)

| Поле | Пример |
|---|---|
| name | «Рынок Панчшанбе» |
| shortName | «Панчшанбе» |
| aliases | «Панджшанбе», «базар», «главный рынок» |
| category | market / cafe / flowers / pharmacy / school / hospital / mall / street / district / landmark / partner |
| lat, lng | 40.28…, 69.62… |
| verified | true после проверки ногами (§G) |
| popularity | 1–100, влияет на порядок в поиске |
| notes | «вход со стороны фонтана», «пин ставить у главных ворот» |

### План 100 точек по категориям

| Категория | Сколько | Примеры/принцип |
|---|---|---|
| Рынки | 5 | Панчшанбе (№1 по popularity) и районные базары |
| Кафе и рестораны | 15 | начиная с будущих партнёров |
| Цветочные | 5 | кандидаты в партнёры |
| Аптеки | 10 | сетевые + по одной на район |
| Школы и вузы | 10 | школы по номерам («Школа №7») + университеты |
| Больницы/поликлиники | 8 | областная, роддом, поликлиники по районам |
| ТЦ и магазины | 10 | торговые центры, крупные магазины |
| Улицы | 15 | главные улицы города (пин — середина/знаковое место) |
| Районы/микрорайоны | 12 | все микрорайоны по номерам + исторические районы |
| Ориентиры | 10 | крепость, парк Камоли Худжанди, вокзал, аэропорт, площади, мосты |

Правило приоритета: сначала то, что люди называют в заказах, а не то, что
легко найти на карте. 20 «верных» точек полезнее 100 сомнительных —
поле `verified` честное.

## G. Как собирать адреса — практический план

1. **Черновик за столом (2–3 часа):** открыть Google Maps/OSM, выписать
   100 точек по плану §F с координатами (long-tap → скопировать координаты).
   verified = false.
2. **Проверка ногами/на электробайке (1 день):** объехать топ-30 по
   popularity; на месте поправить пин (вход, а не центр крыши здания!),
   записать notes («вход со двора», «второй этаж»). verified = true.
3. **Спросить партнёров:** каждый партнёр называет 3–5 мест, «куда чаще
   всего возят» — это готовые популярные точки + их алиасы, как говорят
   клиенты.
4. **Проверить на телефоне:** каждую verified-точку найти поиском в TajGo
   и убедиться, что пин попадает куда надо.
5. **Добавить aliases:** на каждую точку — минимум 2 варианта написания
   (как говорят и как пишут с опечаткой).
6. **Тест поиском:** 20 запросов «как сказал бы клиент» («базар», «седьмая
   школа», «областная») — ≥18 должны найтись с первого раза. Меньше —
   дописать алиасы.

Владелец справочника — владелец TajGo; пополнение — еженедельный ритуал
пилота (из реальных поисковых запросов, которые не нашлись).

## H. Partner places — партнёры как опорные точки

- Точка партнёра (кафе, магазин, цветочный) = запись справочника с
  category=partner, verified=true и заполненными notes (вход, этаж, «спросить
  Фарруха»).
- **Бейдж «Партнёр»** в поиске и на карточке — доверие и реклама партнёру
  одновременно (ему приятно, клиенту надёжно).
- **Выше в поиске**: partner-точки получают приоритет в выдаче — партнёрство
  сразу даёт видимость.
- **Меньше ошибок курьера**: у партнёрской точки пин выставлен на вход и
  есть notes — курьер не ищет дверь. Аргумент для подключения: «ваша точка
  в TajGo выверена — курьеры не будут звонить и спрашивать, где вы».

## I. Что говорить партнёрам про карту

Дословные фразы (дополняют GO_TO_MARKET_STARTER_PACK):

- «TajGo знает локальные места Худжанда короткими названиями — "Панчшанбе"
  достаточно.»
- «Клиенту не надо объяснять адрес по телефону — он ставит точку, курьер её
  видит.»
- «Курьер видит точку и следующий шаг — он не звонит вам спросить дорогу.»
- «Передача подтверждается кодом — "не довезли" исключено.»
- «Ваша точка будет проверенной: с бейджем партнёра, выше в поиске, с
  пометкой, где вход.»

## J. Phone field test — 1 день

Состав: 2 телефона (клиент+админ / курьер), 1 живой курьер, 5 тестовых
заказов, 3 реальные точки из справочника, 1 партнёрская точка.

| # | Тест | Что делать | Успех | Баг |
|---|---|---|---|---|
| 1 | Прогрев | Заказ №1 «вокруг квартала» (A и B рядом) | полный цикл до кода без вмешательства | любое ручное спасение |
| 2 | Реальные точки | Заказы №2–3 на точки из справочника поиском | точки нашлись по коротким именам, пины верные | пин мимо входа >50 м; поиск не нашёл |
| 3 | Партнёрская точка | Заказ №4 от партнёрской точки | бейдж виден, notes у курьера, доехал без звонка | курьер спросил, где вход |
| 4 | Слабый GPS | Часть маршрута заказа №5 — во дворах/в помещении | круг неопределённости, подсказка, кнопки живые | блокировка действий, паника |
| 5 | Обрыв сети | Авиарежим 30 сек посреди этапа B | тихо догнало, заказ цел | красные экраны, потеря статуса |
| 6 | Ошибка адреса | Поиск «то, чего нет» («кафе ромашка») | «Не нашли…» + «Выбрать на карте» → заказ создан пином | тупик без выхода |
| 7 | Админ весь день | Каждый заказ виден на карте города, «GPS N мин назад» живое | тест 5 секунд пройден в любой момент дня | цифры врут, курьер «пропал» на карте |

Итог дня: список багов с приоритетами → Codex; повторный field test после
фиксов, если были баги уровня K-списка (QA §K).

## K. 7-day map pilot

| День | Задача | Критерий дня |
|---|---|---|
| 1 | Карта и GPS: field test §J целиком | все 7 тестов зелёные или баги записаны |
| 2 | Собрать 30 точек ногами (§G шаг 2) | 30 verified, поиск находит все |
| 3 | 5 тестовых доставок разными маршрутами | 5/5 завершены кодом, курьер без чужого навигатора |
| 4 | Подключить 1–2 партнёра (точки + первые заказы) | партнёрская точка в справочнике, тестовый заказ от партнёра прошёл |
| 5 | Реальные маленькие доставки | ≥5 реальных, отзывы записаны дословно |
| 6 | Исправить главные баги (топ-3, не всё) | самая частая проблема закрыта, повторный мини-тест |
| 7 | Решение о расширении | все критерии §B зелёные? → план на 5 курьеров/10 партнёров; нет → второй круг дней 5–6 |

Журнал пилота обязателен: дата, заказ, что сломалось, что сказали люди.

## L. Что нельзя показывать партнёрам

1. Stack trace / англоязычные ошибки на экране.
2. Координаты как основной адрес в карточке.
3. Карту, которая не открылась (без офлайн-запаски-скриншотов).
4. Исчезнувшую или изменившуюся без причины цену.
5. Курьера, который не понял цель и открыл Яндекс.
6. Клиента без ETA («ну наверное скоро»).
7. Пустую админку с нулями.
8. Demo tools без объяснения «это тестовая кнопка».

Любой пункт всплыл на показе — честная фраза: «это тестовый режим, записал,
исправим» — и продолжать по сценарию. Паника хуже бага.

## M. Что можно оставить после пилота

Голосовая навигация · пробки · lane guidance · offline vector maps ·
background navigation (свёрнутое приложение) · коммерческий route-SLA ·
полная база домов с подъездами.

Ничто из этого не мешает возить реальные заказы. Возвращаемся к списку
только после успешного пилота.

## N. Roadmap «лучше навигатора для доставки»

1. **Сейчас (v0.9.1–v1.0):** маршруты по дорогам + turn-by-turn текстом,
   справочник 100 точек с алиасами, партнёрские точки, полевой тест §J,
   производительность карт. Fallback на прямую — навсегда.
2. **Первый пилот:** калибровка порогов (reroute, arrival, скорость ETA) на
   реальных поездках; пополнение справочника из живых запросов; журнал →
   бэклог.
3. **Production:** свой OSRM с SLA, background GPS + батарея, офлайн-кэш
   тайлов города, дорисовка OSM Худжанда, база адресов как продукт.
4. **Масштабирование:** второй город = скопировать плейбук (справочник 100
   точек + партнёры + пилот 7 дней); треки курьеров → свои пробки; батчи
   заказов (B1→B2); такси на той же карте.

## O. Финальный чеклист владельца перед показом

- [ ] APK свежий установлен на оба телефона (из релизной сборки, не из
      случайной).
- [ ] GPS работает на обоих (синяя точка видна, «📡 передаются» у курьера).
- [ ] Карта открывается на Худжанде за ≤3 сек.
- [ ] В справочнике 10–20 популярных точек, поиск «Панчшанбе» находит.
- [ ] Есть заготовленный тестовый заказ-сценарий (A в 100–300 м).
- [ ] Курьер готов: на линии, без активного заказа, телефон заряжен.
- [ ] Клиентский аккаунт готов: отдельный uid, история не пустая.
- [ ] Сценарий показа в голове (DEMO_SCRIPT прогнан вслух накануне).
- [ ] Запасные скриншоты/видео полного цикла — на случай сети/GPS.
- [ ] Фраза на случай сбоя: «тестовый режим, записал, исправим» — и дальше
      по сценарию.

---

**Главное правило пилота:** карта TajGo выигрывает не идеальностью, а тем,
что знает Худжанд и сценарий доставки лучше всех. Каждый день пилота — это
+точки в справочник, +алиасы из реальных запросов, +notes от партнёров.
Это и есть ров, который большим сервисам лень копать.
