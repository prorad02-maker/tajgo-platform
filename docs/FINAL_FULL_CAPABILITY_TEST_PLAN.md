# TajGo 1.5.0 — финальная проверка всех возможностей

Этот план проверяет сборку `1.5.0+30` (`tj.tajgo.app`) от чистого входа до завершённого заказа. Проверка должна выполняться минимум на двух физических Android-устройствах: клиент и курьер. Для admin удобнее использовать третью сессию или отдельный профиль.

## 1. Перед тестом

1. В Firebase Authentication включить Phone и создать отдельные тестовые номера `+992`.
2. Убедиться, что у Android app `tj.tajgo.app` зарегистрированы SHA-1/SHA-256 используемой подписи.
3. Проверить, что `android/app/google-services.json` содержит package `tj.tajgo.app`.
4. Просмотреть локальные `firestore.rules`. Только после разрешения владельца выполнить:

```powershell
firebase.cmd deploy --only firestore:rules
firebase.cmd deploy --only firestore:indexes
```

5. Назначить один UID администратором: `users/{uid}.role = "admin"`.
6. Подготовить минимум один approved courier: в `users/{uid}` должны быть `roles: ["customer", "courier"]`, `courierStatus: "approved"`, `courierOnboardingCompleted: true`.
7. При пустой базе проверить локальные preview-карточки. Затем в админке нажать
   `Опубликовать примеры`: операция создаёт 6 тестовых партнёров и 18 товаров и
   требует admin-role по актуальным Rules.

## 2. Автоматические проверки

Из корня проекта:

```powershell
flutter clean
flutter pub get
flutter analyze
flutter test
flutter build apk --debug --dart-define-from-file=tool/routing.pilot.json
```

Ожидается: analyze без ошибок, все тесты зелёные, APK находится в `build/app/outputs/flutter-apk/app-debug.apk`.

## 3. Account и Phone Auth

- Новый пользователь выбирает роль клиента, вводит номер `+992 XX XXX XX XX`, получает и подтверждает SMS-код.
- После перезапуска остаётся тот же UID, профиль и выбранный режим.
- Неверный/просроченный код показывает понятную ошибку, кнопки не запускают повторные запросы во время загрузки.
- Клиент может начать courier application, сохранить draft, отправить pending; admin может approve/reject; approved пользователь проходит courier onboarding один раз.
- Переключение customer/courier не создаёт второй UID и не удаляет customer-профиль.

## 4. Клиент — своя доставка

1. Нажать `Отправить посылку`.
2. Найти или выбрать A/B на карте, проверить GPS, reverse geocoding, линию маршрута, расстояние и ETA.
3. Ввести допустимую собственную цену и создать заказ.
4. Убедиться, что появился `orders/{orderId}` со статусом `waitingOffers`, `orderType = customDelivery`, координатами и ценой.
5. Проверить список предложений курьеров, выбрать одно, затем отменить другой новый waiting-заказ.

## 5. Клиент — витрина партнёров

1. На главной открыть последовательно `Еда`, `Продукты`, `Цветы`.
2. Проверить открытых/закрытых и выключенных партнёров, карточки, рейтинг, минимум и стоимость доставки.
3. Открыть каталог: недоступный товар нельзя добавить; скрытый товар клиенту не показывается.
4. Добавить товар с единицей `кг`: шаг должен быть `0,5`. Увеличить/уменьшить количество.
5. Попытаться добавить товар другого партнёра: приложение предлагает очистить текущую корзину, молча смешивать партнёров нельзя.
6. Проверить минимальную сумму заказа, итог `товары + доставка`, комментарий и выбор B на карте.
7. Оформить заказ. Перед записью приложение повторно проверяет партнёра, наличие и цену. При изменившейся цене/наличии оформление должно остановиться с ошибкой обновления корзины.
8. В `orders/{orderId}` проверить:
   - `orderType = catalogOrder`, `status = waitingOffers`;
   - `partnerId`, `partnerName`, snapshot `items[]`;
   - `subtotal`, `deliveryFee`, `total`;
   - `price = deliveryFee`, `priceNegotiable = false`;
   - A = адрес партнёра, B = адрес клиента.
9. В tracking должны быть видны партнёр, товары и итог. Корзина после успешного создания очищается.
10. На карте проверить поиск адреса, синий маркер своего GPS, зелёную дорожную
    линию и кнопку обзора всего маршрута.

## 6. Курьер и заказ

1. Войти approved courier, выйти на линию и разрешить точную геолокацию.
2. Проверить обновление `couriers/{uid}.location`, `locationUpdatedAt` и безопасной проекции `courier_public/{uid}`.
3. Waiting-заказ виден только при свежем GPS и расстоянии до A строго меньше 1000 м. Границы: 999 м доступно, 1000 м недоступно.
4. Для custom delivery отправить обычную или повышенную цену. Для catalog order разрешена только фиксированная цена доставки.
5. Клиент выбирает offer. У курьера появляется `activeOrderId`, второй заказ взять нельзя.
6. Проверить карту A/B, живую позицию, маршрут до A, ETA и внешний навигатор.
7. Пройти `accepted → arrivedAtPickupAt → pickedUp → delivered → completed` с гео-гейтом и кодом подтверждения.
8. После pickup цель меняется с A на B. Клиент видит live-маркер, но не приватные поля курьера.
9. После completed очищается `activeOrderId`, а `earningsToday` и `ordersToday` увеличиваются одной транзакцией.
10. Debug-only reset активного заказа виден только в debug и возвращает тестовый заказ в очередь; в release его быть не должно.

## 7. Admin / Dispatch / Marketplace

- Dashboard показывает заказы, активность, курьеров и спорные заказы.
- Courier applications: approve/reject/suspend пишут `admin_logs` и не уничтожают customer-доступ.
- Dispatch map показывает только предусмотренные данные и умеет открыть детали заказа.
- В `Партнёры и товары` admin создаёт/редактирует партнёра и товар, закрывает партнёра, выключает/скрывает товар. Физического удаления из UI нет.
- Карточка партнёра открывает отдельный список ассортимента; JSON import сначала
  проходит dry-run и не пишет данные до подтверждения.
- Неактивный партнёр и скрытый товар не читаются обычным клиентом.
- Изменения создают append-only запись `admin_logs`.
- Обычный customer/courier не может изменить `partners`, `products`, чужие `users`, `couriers`, `orders` или `admin_logs`.

## 8. Негативные и сетевые сценарии

- GPS выключен, permission denied и denied forever дают понятное действие; приложение не зависает.
- Без сети существующие экраны показывают ошибку/пустое состояние и не создают дубликат заказа после повторного нажатия.
- Закрытие партнёра, скрытие товара или изменение цены между корзиной и checkout блокирует старое оформление.
- Два курьера одновременно не могут получить один заказ; один курьер не может иметь два active orders.
- Клиент не читает чужой заказ; курьер не читает чужой назначенный заказ; неизвестные коллекции закрыты.
- Проверить lifecycle background/foreground во время GPS stream, tracking и checkout.

## 9. Критерий готовности

Релиз-кандидат принимается, если автоматические проверки зелёные, оба вида заказов завершаются на физических устройствах, GPS/карты работают на мобильной сети, Rules пропускают только законные операции, а в логах нет необработанных исключений или повторных заказов.

Не считать завершёнными без ручной проверки: реальные SMS, Play Integrity/reCAPTCHA, поведение GPS конкретных моделей телефонов, внешний routing provider, release-подпись и фактический deploy Rules.
