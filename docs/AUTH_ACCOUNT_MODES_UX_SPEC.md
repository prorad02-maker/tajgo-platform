# TajGo v1 — Auth & Account Modes UX Spec

Вход, аккаунт и режимы. Заменяет модель «выберите роль» (текущий RoleScreen
упраздняется) и одиночное поле role как режим. Совместимость со старыми
данными — обязательна (см. §G и HANDOFF этап 14).

---

## A. Первый запуск

После Splash (не авторизован / профиль не создан) — **экран намерения**,
НЕ «выберите роль»:

> **Что вы хотите сделать?**
>
> ┌──────────────────────────────────┐
> │ 📦 Заказать доставку             │
> │ Отправить посылку, документы,    │
> │ еду или покупки                  │
> └──────────────────────────────────┘
> ┌──────────────────────────────────┐
> │ 🛵 Зарабатывать с TajGo          │
> │ Принимать заказы и доставлять    │
> │ по городу                        │
> └──────────────────────────────────┘

Слов «роль», «режим», «аккаунт» на экране нет. Это выбор намерения, который
влияет только на то, куда человек попадёт после входа. Показывается один раз.

## B. Клиентский вход

«Заказать доставку» → телефон → SMS-код → минимальный профиль → CustomerHome.

Тексты:
- Экран телефона: заголовок «Ваш номер телефона», подпись: «По номеру мы
  привяжем ваши заказы и код получения. Никакого спама.» Поле: «+992 …».
  Кнопка: «Получить код».
- Экран кода: «Код из SMS», «Отправили на {номер}. Изменить номер»,
  автосабмит по 6 цифрам, «Отправить ещё раз» (таймер 60 сек).
- Минимальный профиль: одно поле «Как вас называть?» (имя), кнопка «Готово».
  Город = Худжанд по умолчанию, не спрашиваем.

## C. Курьерский вход

«Зарабатывать с TajGo» → тот же телефон/SMS → **анкета курьера**
(COURIER_ONBOARDING §B) → статус «на проверке» → пока проверка идёт,
доступен клиентский режим.

Текст на переходе к анкете: «Осталась короткая анкета — 3 минуты. После
проверки вы сможете выходить на линию.»

Важно: клиентский режим создаётся ВСЕГДА, даже если человек пришёл «в
курьеры»: roles = ["customer"], courierStatus = pending.

## D. Один аккаунт, несколько режимов — правила

1. Каждый пользователь — клиент по умолчанию (roles всегда содержит
   customer).
2. Courier mode добавляется только после одобрения заявки.
3. Повторный аккаунт для второй роли НЕ создаётся — тот же UID.
4. Переключение режимов НЕ требует повторной SMS.
5. Курьер может заказывать доставку (переключился в клиента — заказал).
6. Клиент может подать заявку курьера из профиля в любой момент.
7. Один активный «наём» ролей на телефон: телефон = один основной UID (§J).

## E. Возвращающийся пользователь

Splash определяет маршрут (App Startup Router, HANDOFF этап 3):

- авторизован + профиль есть → открывается **lastMode** (customer/courier);
- экран намерения повторно НЕ показывается никогда;
- courierStatus == pending → клиентский режим + ненавязчивая карточка
  «Заявка на проверке» (тап → статус заявки);
- courierStatus == approved и lastMode == courier → CourierHome;
- телефон не подтверждён (легаси-аноним) → мягкий баннер «Войдите по
  телефону» в клиентском режиме (не стена).

## F. Переключение режимов

В Профиле:
- у клиента с approved-курьером: «🛵 Перейти в режим курьера»;
- у курьера: «👤 Перейти в режим клиента»;
- у клиента без заявки: «🛵 Зарабатывать с TajGo» (→ анкета).

Переключение мгновенное (замена корневого экрана), lastMode пишется в
профиль. Тумблер «на линии» остаётся внутри курьерского режима и никак не
связан с переключением режимов.

## G. Данные

```
users/{uid}:
  phoneNumber: "+992..."
  displayName: "Фаррух"
  roles: ["customer"]            // + "courier" после одобрения
  lastMode: "customer"           // "customer" | "courier"
  courierStatus: "none"          // none|draft|pending|approved|rejected|suspended
  phoneVerified: true
  profileComplete: true
  role: "customer"               // LEGACY, поддерживаем на чтение и запись
  city, language, createdAt, updatedAt — как сейчас
```

Правила совместимости:
- `role` (строка) НЕ удаляется: admin определяется по `role == "admin"`
  (как в задеплоенных правилах и админке) — это остаётся источником истины
  для admin. Для customer/courier `role` пишется как lastMode (зеркало),
  но ЧИТАЮТСЯ режимы из `roles[]`.
- Старый документ без `roles` → мигрируется при первом входе:
  roles = [role == 'courier' ? 'customer','courier' : 'customer'],
  courierStatus = существует couriers/{uid} ? approved : none (легаси-курьеры
  не проходят повторную проверку).
- couriers/{uid} и courier_public/{uid} — без изменений структуры.

Статусы courierStatus: none → draft → pending → approved | rejected;
approved → suspended → approved. Матрица — в COURIER_ONBOARDING.

## H. Anonymous fallback

**Debug/demo-сборка:** fallback допустим; на экране обязательно видно, что
это тестовый режим (бейдж «Тест · демо-вход» на главной), реальные партнёры
такого не видят.

**Release:** anonymous fallback ВЫКЛЮЧЕН по умолчанию. Без подтверждённого
телефона нельзя создать реальный заказ и выйти на линию (кнопки ведут на
вход по телефону).

Механизм: **feature flag в коде** (константа/конфиг `allowAnonymousDemo`,
true только в debug). Никакого Firebase deploy в рамках этой задачи;
отключение Anonymous-провайдера в консоли — отдельное решение владельца
после миграции (FIREBASE_PRODUCTION_SETUP §1).

## I. Ошибки Phone Auth — тексты

| Ситуация | Текст |
|---|---|
| Неверный номер | «Проверьте номер. Нужен формат +992 XX XXX XX XX.» |
| SMS не пришёл | «Код не пришёл? Проверьте номер и запросите новый — иногда SMS задерживается на минуту-две.» |
| Неверный код | «Неверный код. Проверьте SMS и попробуйте ещё раз.» |
| Слишком много попыток | «Слишком много попыток. Подождите немного и попробуйте снова.» |
| Нет интернета | «Нет связи. Проверьте интернет и повторите.» |
| Номер уже связан | «Этот номер уже привязан к аккаунту TajGo. Входим в него…» (см. §J) |
| Сессия истекла | «Срок действия кода истёк. Запросите новый.» |
| Phone Auth не настроен | «Вход по SMS временно недоступен. Попробуйте позже.» (владельцу в лог: operation-not-allowed) |

Существующий `PhoneAuthFailure.fromFirebase` уже близок — привести тексты
к таблице.

## J. Account conflict — безопасное поведение

- Один телефон = один основной UID. Точка.
- Anonymous UID при вводе телефона: сначала пытаемся `linkWithCredential`
  (текущее поведение — сохранить).
- `credential-already-in-use` (телефон уже занят другим UID): НЕ создавать
  дубликат и НЕ сливать данные автоматически. Показать **отдельный экран**:

  > «Этот номер уже привязан к аккаунту TajGo.
  > Войти в него? Текущие тестовые данные этого устройства останутся
  > в старом профиле.»
  > [Войти в существующий аккаунт] [Отмена]

  «Войти» → signInWithCredential в существующий UID (текущее поведение).
- Существующие заказы НИКОГДА не удаляются и не переписываются при
  конфликте. Автоматическое объединение двух профилей запрещено — только
  вручную владельцем (после v1 — серверный процесс).

## K. Выход и удаление

- **Выход**: пункт в Профиле → подтверждение («Выйти из аккаунта? Заказы
  сохранятся — войдите этим же номером.») → signOut → экран намерения (§A).
- Локальные данные: lastMode и кэш сбрасываются; Firestore не трогается.
- Курьер на линии при выходе → сначала снимается с линии (online=false),
  потом signOut.
- **Удаление аккаунта**: в v1 — пункт «Удалить аккаунт» открывает экран
  «Напишите нам, удалим в течение N дней» (контакт владельца). Кнопки,
  удаляющей Firestore-данные одним нажатием, НЕТ — удаление только
  серверным процессом после v1 (иначе осиротеют заказы/статистика).
