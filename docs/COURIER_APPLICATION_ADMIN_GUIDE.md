# TajGo — руководство администратора по заявкам курьеров

## Доступ

Раздел «Заявки курьеров» находится на AdminHome. В release доступ к админским экранам должен иметь только аккаунт с серверно назначенной ролью `admin`. Debug-интерфейс не заменяет реальные права Firestore.

## Очередь заявок

Список показывает:

- имя и телефон;
- транспорт;
- дату отправки;
- текущий статус;
- счётчик заявок `pending` на AdminHome.

Фильтры: pending, draft, approved, rejected, suspended и все.

## Проверка заявки

Перед решением сопоставьте:

1. профиль и номер телефона;
2. выбранный транспорт;
3. тип и номер документа;
4. фото или отметку о проверке при личной встрече;
5. город, район и доступность;
6. согласия с правилами и обработкой данных;
7. историю предыдущих решений.

В текущей сборке Storage выключен, поэтому отсутствие загруженного файла не является технической ошибкой: приложение явно показывает `personalMeeting`. До запуска реальной загрузки документ нужно сверять при личной встрече.

## Действия

### Approve

Используйте только после проверки данных. Операция одной Firestore-транзакцией:

- переводит заявку и пользователя в `approved`;
- сохраняет `customer` и добавляет `courier` без дублей;
- создаёт/обновляет private/public профиль курьера;
- сохраняет статистику и `activeOrderId` существующего профиля;
- оставляет новый профиль offline;
- пишет `approveCourier` в `admin_logs`.

Курьер увидит экран «Вы приняты», затем одноразовый onboarding.

### Reject

Причина обязательна. Пишите конкретно и без персональных данных, например: «Номер документа не читается». Пользователь остаётся клиентом, увидит причину и сможет исправить анкету.

### Suspend

Причина обязательна. Курьер немедленно становится offline, режим курьера блокируется, клиентский доступ сохраняется. `activeOrderId` специально не очищается: если есть активная доставка, администратор должен решить её отдельно.

### Restore

Возвращает `approved`, но не выводит курьера на линию автоматически. Курьер сам включает online после входа в режим.

## Журнал действий

Каждое решение создаёт неизменяемую запись `admin_logs`:

```text
action
targetUid
adminUid
reason (когда нужен)
createdAt
```

Если решение не завершилось полностью, транзакция откатывается вместе с журналом. Не исправляйте статусы вручную в нескольких документах Firestore: это может рассинхронизировать аккаунт и заявку.

## Безопасная эксплуатация

- не отправляйте паспортные данные в чаты и логи;
- не добавляйте реальные документы в тестовую среду;
- не назначайте admin из мобильного клиента;
- до deploy отдельно проверьте diff `firestore.rules`;
- тестируйте reject/suspend только на тестовом курьере;
- после suspend проверьте активную доставку в админке;
- при `permission-denied` не ослабляйте правила: сначала убедитесь, что аккаунт действительно admin и новая версия rules развёрнута.
